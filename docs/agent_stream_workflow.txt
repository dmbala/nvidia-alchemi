How the agent runs the HT screen and streams results to active learning
=======================================================================

The agent has two concurrent responsibilities that share ONE directory
(runs/screen/results/) as a producer-consumer buffer:

  HT SCREEN  (producer)          ─▶  runs/screen/results/  ─▶  ACTIVE LEARNING  (consumer)
  continuous, never idles             append-only stream        fires when N new results land

Section 2 — What the agent does on the HT screen side
-----------------------------------------------------

  One tick (HT portion):

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  python -m orchestrator.ht_loop --action tick                           │
  │    (called by agent via Bash tool)                                      │
  │                                                                         │
  │  IF no SLURM job yet:                                                   │
  │    • sbatch --array=0-N run_screening.slurm                             │
  │    • record job_id in screen_state.json                                 │
  │                                                                         │
  │  ELSE for every chunk:                                                  │
  │    • sacct state + filesystem check                                     │
  │    • if result CSV present  → mark done                                 │
  │    • if sacct COMPLETED but no CSV → mark anomaly (pipeline bug)        │
  │    • if OOM   → queue retry with 2× mem (up to 3 attempts)              │
  │    • if time  → queue retry with 2× walltime                            │
  │    • if perma → mark failed_permanent (stream continues without it)     │
  │                                                                         │
  │  emit retries as new --array=<chunk_ids> submission with offset         │
  │    (windowed if chunk_id ≥ MaxArraySize=10000)                          │
  └─────────────────────────────────────────────────────────────────────────┘

  The agent layer on top reads the pending_decisions emitted here and
  writes a natural-language summary ("chunk 3 recovered after OOM retry",
  "chunk 7 is an anomaly — logs show RDKit parse error on malformed SMILES").

  Screen results accumulate append-only:

      runs/screen/results/
        result_0000.csv   ┐
        result_0001.csv   │  each chunk completion adds one file
        result_0002.csv   │  files are NEVER deleted by the orchestrator
        result_0003.csv   │  (only merged-back verify adds more via symlink)
        result_0004.csv   │
        ...               ▼
        result_verify_XX.csv  ← symlinked from runs/al/iter_N/verify_results/


Section 3 — How the stream flows into active learning
-----------------------------------------------------

  Each tick, the AL side checks:

     count(runs/screen/results/*.csv) − last_snapshot_size  ≥  trigger_threshold?

  YES → start a new AL iteration. NO → stay idle, wait for more results.

  The CRUCIAL DETAIL is the snapshot: when an iter fires, the agent
  materializes a read-only view of the stream at that exact moment so
  training is reproducible even as the stream keeps growing.

  ┌─────────────────────────────────────────────────────────────────────────┐
  │                                                                         │
  │   runs/screen/results/                    runs/al/iter_NNN/             │
  │      result_0000.csv ◀────┐                 screen_snapshot/            │
  │      result_0001.csv ◀────┼── symlinks ──▶   result_0000.csv ─▶ (link)  │
  │      result_0002.csv ◀────┤                   result_0001.csv ─▶ (link)  │
  │      result_0003.csv ◀────┤                   result_0002.csv ─▶ (link)  │
  │      result_0004.csv ◀────┘                   ...                       │
  │      result_0005.csv         ← newer results appear later;              │
  │      result_0006.csv           they're NOT in this iter's snapshot      │
  │      ...                       (they'll be in iter N+1's)               │
  │                                                                         │
  │                                              snapshot.txt   (manifest)  │
  │                                                                         │
  └─────────────────────────────────────────────────────────────────────────┘

  Training sees a stable directory; screen keeps growing behind the scenes.

Section 5 — The key insight
---------------------------

  Screen and AL share ONE stream directory but decouple in time:

      ┌──────────────────────────────────────────────────────────┐
      │  SCREEN writes continuously (append-only)                │
      │      ▲                                                   │
      │      │ +result_NNNN.csv per chunk                        │
      │                                                          │
      │      runs/screen/results/                                │
      │                                                          │
      │      │  +result_verify_NNNN.csv per merged verify        │
      │      ▼                                                   │
      │  AL snapshots periodically (read-only views)             │
      │      symlinks only, so no copies, no drift,              │
      │      reproducible training sets                          │
      └──────────────────────────────────────────────────────────┘

  The agent's tick is the ONE place both sides converge:
     1. advance screen (submit/poll/retry)
     2. check AL trigger
     3. if an AL iter is active, advance its current phase
     4. summarize in ≤4 lines of text

  Screen never pauses for AL. AL iter N+1 naturally gets more data
  than iter N because the stream grew. Verified candidates loop back
  as training data for future iters — the feedback loop that makes
  the active-learning part of "active learning".
